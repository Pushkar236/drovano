# Drovano Architecture Research Report — Stack Decisions for an AI-Native Multi-Tenant B2B Business OS

**Research date: 2026-07-06.** All version numbers verified against npm registry / GitHub releases / official docs on this date unless flagged. Confidence markers: **[H]** primary/official source, **[M]** credible secondary, **[L]** weak/aggregator. A refuted-and-unverified-claims appendix is at the end. Intended as raw ADR input.

## 0. Recommended stack at a glance

| Layer | Pick | Runner-up / escape hatch |
|---|---|---|
| Monorepo | pnpm 10 workspaces + Turborepo 2.x + TS project references | Nx (only at 50+ engineers / DTE need) |
| Frontend | Vite 8 + React 19.2 + TanStack Router v1 + TanStack Query v5, SPA-first | TanStack Start (RC) when SSR needed; Next 16 for marketing site only |
| Client data layer | TanStack Query → TanStack DB collections (optimistic txns) | ElectricSQL shapes per-collection at scale; re-evaluate Zero ~1.10+ |
| Backend | Modular monolith; Hono 4.x (or Fastify 5) | Extract workers/services only on divergent scaling needs |
| API | tRPC v11 internal; REST + OpenAPI from Zod v4 (`@hono/zod-openapi`) + Speakeasy SDKs public; MCP server public | oRPC 1.x as single-system alternative |
| Database | Postgres 18, shared schema + `tenant_id` (uuidv7) + RLS, Neon/Supabase managed | Bridge to silo DBs for enterprise tier; Citus/PlanetScale Postgres for scale-out |
| ORM | Drizzle (0.45.x now, 1.0 at RC) + Kysely for reporting SQL | Prisma 7 (legitimate, weaker RLS story) |
| Vectors | pgvector 0.8.x (halfvec + HNSW + iterative scans) | Turbopuffer at ~10M+ vectors |
| Jobs / agents runtime | Trigger.dev v4 (self-host exit hatch); pg-boss for trivial MVP jobs | Temporal at genuine workflow-infra scale |
| Auth | better-auth 1.6.x self-hosted + organization plugin | WorkOS SSO/SCIM bolt-on per enterprise connection if better-auth SAML proves green |
| AI layer | Vercel AI SDK v7; hand-rolled/`ToolLoopAgent` loops per Anthropic guidance; Claude Agent SDK for heavyweight workers | Mastra if team wants batteries-included |
| RAG | Contextual retrieval + hybrid (BM25+dense) + rerank (Cohere 3.5/4.0), retrieval-as-agent-tool | — |
| Infra | Vercel (frontend) + Render containers + Neon PG; OTel 2.x + Sentry (+ Axiom for volume logs) | SST v3 → AWS at compliance/enterprise stage; OpenTofu for raw IaC |
| Realtime | Own thin WS service + Redis pub/sub (invalidation); PartyKit/Durable Objects for presence | CDC via ElectricSQL shapes as read-path sync hardens |

## 1. Monorepo tooling

**State (registry/GitHub-verified 2026-07-06) [H]:** `turbo` **2.10.3** (Jul 3, 2026; ~16.3M weekly downloads); `nx` **23.0.1** (Nx 23 released Jun 16, 2026; ~8.9M/wk); Bun **1.3.14** (May 2026; Bun 1.3 added workspace catalogs, isolated installs default, security-scanner API — https://bun.com/blog/bun-v1.3). No Bun 2.0 exists despite blogspam.

**Turborepo** ships fast under Vercel: 2.5 sidecar tasks/`turbo.jsonc` → 2.6 microfrontends + **stable Bun PM support** → 2.7 devtools → 2.8 git worktrees + AI/Agent Skill → 2.9 ("up to 96% faster", deprecations telegraphing a 3.0) → 2.10 composable `--affected`+`--filter`, cache eviction (https://turborepo.dev/blog) [H]. `turbo watch` since 2.0; **Boundaries** (module-constraint enforcement via tags) still experimental (https://turborepo.dev/docs/reference/boundaries) [H]. Free hosted Vercel remote cache + documented self-host protocol [M].

**Nx** is technically the most powerful (generators, constraints, distributed task execution; good TS-project-references push via `nx sync` — https://nx.dev/blog/new-nx-experience-for-typescript-monorepos [H]) but carries two real negatives: (1) the **Aug 26, 2025 "s1ngularity" supply-chain attack** — malicious `nx` versions exfiltrated GitHub/npm/SSH/AI credentials, ~2,349 secrets leaked, 190+ orgs affected; first attack weaponizing installed AI CLIs (https://nx.dev/blog/s1ngularity-postmortem, https://www.wiz.io/blog/s1ngularity-supply-chain-attack) [H]; (2) monetization whiplash — Powerpack paywalled self-hosted caching (Sep 2024), backlash, free plugins restored in 20.8, then reportedly deprecated again May 2026 citing CVE-2025-36852, pushing Nx Cloud (https://nx.dev/blog/introducing-nx-powerpack; https://emilyxiong.medium.com/exploring-of-nx-self-hosted-cache-5bc39bd2ed7f) [M — single secondary source for the 2026 re-deprecation].

**What reference TS-SaaS teams run (direct repo inspection, 2026-07-06) [H]:** Cal.com (yarn 4 + Turborepo 2.7), Dub (pnpm 9 + Turborepo), Midday (**Bun 1.3.11 as packageManager** + Turborepo 2.9), Supabase (pnpm 10 + Turborepo 2.9), Trigger.dev (pnpm 10 + Turborepo) — **5/5 Turborepo, 0/5 Nx, 3/5 pnpm** (e.g. https://github.com/midday-ai/midday/blob/main/package.json, https://github.com/supabase/supabase/blob/master/package.json). State of JS 2024: pnpm most-used workspace tool (https://2024.stateofjs.com/en-US/libraries/monorepo_tools/) [H]; 2025 edition page not published.

**Recommendation:** **pnpm 10 workspaces (with catalogs) + Turborepo 2.x + TypeScript project references.** Boring, dominant in exactly this niche, free self-hosted caching, Boundaries gives cheap architecture enforcement for the modular monolith. Use Bun tactically (`bun test`, scripts) but keep Node LTS runtime + pnpm lockfile of record; Midday proves Bun-as-PM works if speed ever matters. Adopt Nx only if you later need distributed task execution at 50+ engineers. Watch for Turborepo 3.0 (deprecations already landing in 2.9).

## 2. Frontend for a Linear/Attio-class app

### Framework state (all [H] unless noted)

- **Next.js 16** (Oct 21, 2025; point releases through 16.3 — https://nextjs.org/blog/next-16, https://nextjs.org/blog/next-16-3-instant-navigations). Turbopack now default bundler; Cache Components (PPR + `use cache`); `proxy.ts` replaces middleware. React **19.2** current stable (Oct 1, 2025; `<Activity/>` — https://react.dev/blog/2025/10/01/react-19-2). Persistent, documented awkwardness for pure-SPA/dashboard use (https://github.com/vercel/next.js/discussions/60365, https://github.com/vercel/next.js/discussions/91475) and an active anti-Next sentiment cycle around App Router complexity + Vercel billing [M].
- **Vite 8.0** (stable Mar 12, 2026) with **Rolldown as the unified Rust bundler** (10–30x build speedups claimed; `rolldown-vite` merged into core and archived; ~65M weekly downloads — https://vite.dev/blog/announcing-vite8). **TanStack Router v1** mature (`1.170.x`, de facto type-safe SPA router — https://github.com/TanStack/router/releases). **TanStack Query 5.101.x**. **TanStack Start is still v1 RC, not GA** — official site verified live this week shows RC badge (https://tanstack.com/start/latest); a third-party "1.0 shipped March 2026" claim conflicts and was discarded [H].
- **React Router 7** framework mode is the stable Remix successor (https://remix.run/blog/react-router-v7); **Remix 3 is a different product — no React (Preact fork), beta preview Apr 30, 2026, explicitly not production ready** (https://remix.run/blog/remix-3-beta-preview) [H]. Strategically ambiguous team direction; avoid for a React startup.

### What the reference class actually built [H/M]

None are Next/RSC apps. **Linear:** React SPA + Electron, MobX in-memory object graph, IndexedDB, GraphQL mutations/bootstrap, WebSocket delta sync (CTO-endorsed reverse engineering: https://github.com/wzhudev/reverse-linear-sync-engine; https://performance.dev/how-is-linear-so-fast-a-technical-breakdown). **Attio:** bespoke client-heavy React SPA, own reconciler + virtualized data list (https://attio.com/engineering/blog). **Notion:** React SPA + **WASM SQLite over OPFS** for local cache, 20–33% faster navigations (https://www.notion.com/blog/how-we-sped-up-notion-in-the-browser-with-wasm-sqlite). **Superhuman:** offline-first idempotent modifier-queue optimistic architecture (https://blog.superhuman.com/architecting-a-web-app-to-just-work-offline-part-1/). The pattern: **the data layer is the product; the meta-framework is irrelevant.**

### Sync engines — how real in 2026?

Materially more real than 2024, but adoption posture matters:

- **Zero (Rocicorp):** **1.0 stable June 8, 2026; currently 1.7** (https://zero.rocicorp.dev/docs/release-notes, https://www.infoq.com/news/2026/06/zero-version-1/) [H]. Apache-2 client+server, self-hostable `zero-cache` over a Postgres read replica; monetizes via hosted service (invite-only per roadmap). Known 1.0 gaps: Postgres-only, no SSR, weak rejected-write handling, 232KB gzipped client. **No named production customers found** [L on production evidence]. Replicache is maintenance-mode, Reflect shut down (https://replicache.dev/, https://rocicorp.dev/blog/retiring-reflect) [H].
- **ElectricSQL:** read-path sync of Postgres "shapes" over plain HTTP (CDN-cacheable), writes stay in your API (https://electric-sql.com/blog/2024/11/21/local-first-with-your-existing-api). **Electric Cloud GA; named production users Google, Supabase, Trigger.dev ("millions of updates/day"), Otto, Doorboost** (https://electric-sql.com/sync) [H, first-party testimonials] — the strongest production evidence of any sync engine. Co-develops PGlite with Supabase sponsorship (https://supabase.com/partners/integrations/electricsql).
- **TanStack DB:** **beta**; typed collections + differential-dataflow live queries + optimistic transactions with rollback; backend-agnostic loaders (Query/REST/tRPC/Electric/PowerSync); **v0.6 (Mar 25, 2026) added SQLite persistence + nested `includes`** — Electric calls it "app-ready" (https://tanstack.com/db/latest, https://electric-sql.com/blog/2026/03/25/tanstack-db-0.6-app-ready-with-persistence-and-includes) [H]. Built by ex-Electric people, deliberately removable.
- **LiveStore** (Schickling): elegant event-sourced SQLite model, **pre-1.0** (last confirmed v0.4.0-dev, Dec 2025), smallest production footprint; event-log-as-schema is a heavy modeling commitment (https://github.com/livestorejs/livestore, https://expo.dev/blog/local-first-application-development-with-livestore) [M].
- **Convex** = full BaaS with sync at core — max lock-in, replaces your backend (https://www.convex.dev/sync) [M]. **Yjs** remains the production CRDT standard for collaborative documents; **Automerge 3.0** (Aug 2025) cut memory 10–100x (https://automerge.org/blog/automerge-3/) [H]. CRDTs are for doc surfaces, not relational records.

**Verdict:** do **not** bet the core data layer on a sync engine yet, and do **not** hand-roll a Linear-style engine in 2026. Posture: (1) structure the client around **TanStack DB collections from day one** (beta risk contained — it's a client library over your own API), starting with Query/REST-backed collections + WebSocket invalidation → immediate optimistic mutations with rollback and live queries, zero server lock-in; (2) adopt **Electric shapes per-collection** when read-path sync earns its keep (activity feeds, live lists) — most proven, least invasive; (3) re-evaluate **Zero** at ~1.10+ once production logos and SSR/error-handling gaps close; (4) **Yjs** only for the collaborative-docs feature. This mirrors the Linear/Notion architecture without their build-from-scratch tax.

**Framework recommendation:** **Vite 8 + React 19.2 + TanStack Router + TanStack Query v5, SPA-first**; marketing site separate (Next 16 or Astro). TanStack Start (same router underneath) is a cheap later migration if in-app SSR/server functions become necessary. Next App Router optimizes the wrong axis for this product class and its server-centric data flow fights an optimistic/sync data layer.

## 3. Backend architecture + API layer

**Modular monolith vs microservices:** consensus is firmly modular-monolith-first. Shopify's canonical 2.8M-line modular monolith stance (https://www.shopify.com/enterprise/blog/monolithic-to-microservices) [H]; AWS Prescriptive Guidance warns premature decomposition is costly [M]; widely-circulated "42% consolidating / 60% regret" stats trace only to low-quality aggregators — **treat as unverified** [L]. Key insight for Drovano: 100k+ tenants is a **data-layer problem** (partitioning, pooling, RLS), not a service-count problem. Pattern: modular monolith with enforced package boundaries (Turborepo Boundaries + TS project references) + extracted async workers; extract services only on divergent scaling/compliance needs.

**Frameworks (npm-verified) [H]:** Hono **4.12.27** (~47M wk downloads; runs Node/Bun/Workers/Deno; first-class `@hono/zod-openapi`, `@hono/trpc-server`); Fastify **5.10.0** (mature, best pure-Node plugin/encapsulation model); NestJS **11.1.27** (no v12; heavy DI ceremony — Cal.com chose it for public API v2: https://github.com/calcom/cal.com/blob/main/apps/api/v2/package.json); Express **5.2.1** (maintenance-mode ecosystem, not greenfield). **Midday's production API is Hono + tRPC + zod-openapi on one server** (https://github.com/midday-ai/midday/blob/main/apps/api/package.json) [H].

**API layer (npm-verified) [H]:**
- **tRPC `@trpc/server` 11.18.0** — v11 stable since Mar 2025, battle-tested (Cal.com maintains a whole trpc workspace), weak OpenAPI story (trpc-openapi unmaintained — https://blog.logrocket.com/trpc-vs-orpc-type-safe-rpc/) [M].
- **oRPC 1.14.6** — tRPC DX + first-class OpenAPI from the same procedures; **615k wk downloads ~15 months post-1.0** — real traction (https://orpc.dev/blog/v1-announcement, https://www.infoq.com/news/2025/12/orpc-v1-typesafe/) [H].
- **Zod 4.4.3** (v4 stable since Jul 2025, 211M wk downloads) [H]. **ts-rest is stale** (~16 months no publish) — avoid [H].
- **GraphQL:** declining as a default for new single-client products; 0/5 reference SaaS repos use it for core APIs [H by repo inspection]; job-listing decline figures are aggregator-sourced [L].
- **SDK generation shifted hard in 2026:** **Stainless acquired by Anthropic (~$300M, May 18, 2026), hosted generator winding down — do not build on it** (https://techcrunch.com/2026/05/18/anthropic-has-acquired-the-dev-tools-startup-used-by-openai-google-and-cloudflare/) [H]; **Fern acquired by Postman (Jan 8, 2026), continuing** (https://blog.postman.com/postman-acquires-fern/) [H]; **Speakeasy** the remaining major independent — Dub's official Node SDK is Speakeasy-generated (https://github.com/dubinc/dub-node) [H].
- **Effect-TS:** `effect` 3.21.4, v4 in beta (rewritten runtime — https://www.infoq.com/news/2026/04/effect-v4-beta/) [M]. Real power (typed errors, DI), but total programming-model buy-in mid-major-transition; wrong for hiring velocity now, fine tactically later.

**Recommendation:** Modular monolith on **Hono** (portable across Node/Workers, zod-native OpenAPI, proven tRPC cohabitation at Midday); Fastify 5 equally defensible if you want its plugin encapsulation. **Split internal vs public API** as every reference company converged on: tRPC v11 for dashboard↔backend (zero contract drift), REST + OpenAPI generated from Zod v4 for the public API, **Speakeasy** for customer SDKs. oRPC is the forward bet if you want one system for both surfaces; tRPC is the battle-tested one. Plus an MCP server on the public API (see §6).

## 4. Database layer

**Context:** **PostgreSQL 18** current (Sep 25, 2025; async I/O up to 3x read throughput; native `uuidv7()` — ideal tenant-scoped PKs; PG19 beta1 out — https://www.postgresql.org/about/news/postgresql-18-released-3142/) [H]. Market consolidation is now a selection criterion: **Databricks bought Neon (~$1B, May 2025** → Lakebase GA Feb 2026; Neon.com continues, storage price cut to $0.35/GB-mo) (https://www.techtarget.com/searchdatamanagement/news/366623864/Databricks-adds-Postgres-database-with-1B-Neon-acquisition, https://www.infoq.com/news/2026/02/databricks-lakebase-postgresql/) [H]; **Snowflake bought Crunchy Data (~$250M, Jun 2025)** (https://www.cnbc.com/2025/06/02/snowflake-to-buy-crunchy-data-250-million.html) [H]; **PlanetScale for Postgres GA Sep 22, 2025** (Vitess-lineage sharded PG — https://planetscale.com/blog/planetscale-for-postgres-is-generally-available) [H]. Supabase remains independent.

### Multi-tenancy pattern at 100k+ tenants

- **Shared schema + `tenant_id` + RLS (pool model) is the recommended default** and AWS's prescribed pattern for pooled tenancy (https://docs.aws.amazon.com/wellarchitected/latest/saas-lens/silo-pool-and-bridge-models.html, https://docs.aws.amazon.com/prescriptive-guidance/latest/saas-multitenant-managed-postgresql/rls.html) [H]. Policy design consensus: `tenant_id = current_setting('app.current_tenant_id')::uuid` set via `SET LOCAL` inside a transaction — the exact shape compatible with PgBouncer/pgcat transaction-mode pooling with no cross-connection leakage (https://www.permit.io/blog/postgres-rls-implementation-guide, https://www.pgbouncer.org/config.html) [H]. If using JWT-claim functions, wrap in an initplan `(select ...)` so evaluation is once-per-query — Supabase's own guidance; fixes 100x pathologies (https://supabase.com/docs/guides/troubleshooting/rls-performance-and-best-practices-Z5Jjwv) [H].
- **Gotchas (all verified):** composite indexes with `tenant_id` leading are the #1 perf factor (~0.3ms policy overhead at 50M rows/10k tenants when indexed — https://oneuptime.com/blog/post/2026-01-25-row-level-security-postgresql/view [M]); non-LEAKPROOF functions block predicate pushdown; table-owner/superuser silently bypass RLS — use a non-owner app role + `FORCE ROW LEVEL SECURITY`; RLS doesn't cover matviews/exports/background jobs (https://www.bytebase.com/blog/postgres-row-level-security-footguns/) [H]. Done right, overhead ~1–5% [M].
- **Honesty note:** no named public case study of **RLS specifically at 100k+ tenants** was found [L]; the pattern (shared schema + tenant_id) indisputably scales — that's just rows, and Citus documents "millions of tenants" for tenant_id sharding (https://www.citusdata.com/use-cases/multi-tenant-apps/). Counter-view exists: PlanetScale's "RLS sounds great until it isn't" argues app-layer scoping + testing at scale (https://planetscale.com/blog/rls-sounds-great-until-it-isnt) [H that the argument exists]. Failure stories involve complex policies and missing indexes, not the simple single-GUC-equality design.
- **Schema-per-tenant: disqualified at this scale** — pg_catalog bloat, O(N) migrations (45+ min deploys reported), practical ceiling low-thousands of tenants (https://planetscale.com/blog/approaches-to-tenancy-in-postgres, https://oneuptime.com/blog/post/2026-01-25-multi-tenant-schemas-postgresql/view) [H].
- **Database-per-tenant: viable only as an enterprise *tier*** via a control-plane/catalog DB (Neon documents the pattern + scale-to-zero economics: https://neon.com/use-cases/database-per-tenant, https://neon.com/blog/control-planes-for-database-per-user-in-neon) [H]. **Nile** (tenant-virtualized Postgres, built-in `tenants` table, per-tenant backups — https://www.thenile.dev/) is purpose-built but still public-preview/seed-stage — interesting, not a foundation bet [L on GA].
- **Scale-out exits:** Citus 13.x (https://docs.citusdata.com/en/stable/articles/sharding_mt_app.html) or PlanetScale Postgres. **The only MVP choice preserving every exit is shared-schema + tenant_id on every row.**

### ORMs (npm-verified) [H]

- **Drizzle:** stable **0.45.2**; **1.0.0-rc.4** (Jun 27, 2026) — v1 ships Relational Queries v2 (`defineRelations()`); **the only major TS ORM with RLS policies as schema-code** (`pgPolicy`, Neon/Supabase auth integrations — https://orm.drizzle.team/docs/latest-releases, https://orm.drizzle.team/docs/v0-v1-changes). Crossed Prisma in weekly downloads during 2025 [M]. Production: Midday (`^0.45.1`), OpenStatus — repo-verified.
- **Prisma 7.8.0:** Rust engine gone — TS/WASM query compiler, ~90% smaller bundles, claimed ~3x faster (vendor benchmark) (https://www.prisma.io/blog/from-rust-to-typescript-a-new-chapter-for-prisma-orm) [H feature, M perf]. Company monetizes Prisma Postgres; ORM is the funnel. Established users (Dub, Cal.com, Documenso) are still on Prisma **6** — v7 migration non-trivial [H]. No first-class RLS; tenancy via client-extension `SET LOCAL` wrappers.
- **Kysely 0.29.3:** the SQL-first typed escape hatch; pairs well with either (Documenso pattern) [H].

**Recommendation:** **Drizzle** (pin 0.45.x, jump to 1.0 at GA) for schema-as-code RLS + SQL proximity + lightweight workers; Kysely for gnarly reporting queries.

### pgvector

Current **0.8.4** (tag-verified; release date unconfirmed) [H/M]. The 0.8 features are exactly the multi-tenant fix: **iterative index scans** (`hnsw.iterative_scan`) solve filtered-search-returns-too-few-rows for `WHERE tenant_id = X ORDER BY embedding <=> q`; **halfvec** halves HNSW index size (https://www.postgresql.org/about/news/pgvector-080-released-2952/) [H]. RLS composes with vector queries (Nile demonstrates tenant-isolated embeddings on this model — https://www.thenile.dev/docs/extensions/vector). Ops gotchas: HNSW handles deletes poorly → periodic `REINDEX CONCURRENTLY` under heavy re-ingest; partial HNSW indexes for whale tenants (https://github.com/pgvector/pgvector/issues/575) [M-H]. Comfortable to ~10–50M vectors / moderate QPS [M — vendor-influenced thresholds]. **Escape hatch: Turbopuffer** — namespace-per-tenant with no practical namespace cap; Cursor (80M+ namespaces) and Notion (10B+ vectors, −80% cost) are the marquee migrations; trade-off 300–800ms cold queries (https://turbopuffer.com/customers/cursor) [M — vendor-sourced]. Qdrant best filtered-search price/perf self-hosted mid-scale; Pinecone namespace caps bite multi-tenant SaaS [M].

**Recommendation:** embeddings in separate per-domain tables on pgvector (halfvec + HNSW + iterative scans + tenant_id-leading B-trees); plan the Turbopuffer namespace-per-tenant migration at ~10M+ vectors or when hybrid/native BM25 requirements harden. Both the DB and AI research threads converged on this independently.

### Jobs / background work (npm-verified) [H]

- **pg-boss 12.25.x** — very active; partitioned job tables, DLQs, cron; right when Postgres is your only stateful dependency; ceiling ~high-thousands jobs/sec (https://github.com/timgit/pg-boss). **Graphile Worker 0.17.2** — official benchmark ~196k jobs/sec processed (https://worker.graphile.org/docs/performance). Note: poll-based queues keep scale-to-zero Postgres awake (cost interplay with Neon).
- **BullMQ 5.79.x** — mature, OTel support since 5.71; adds Redis SPOF; queueing, not durable multi-step execution (https://docs.bullmq.io/changelog).
- **Trigger.dev v4 GA** — SDK 4.5.0; free 10k runs/mo, Pro $50/mo (250k), Team $200/mo (1M); **officially supported self-hosting (Docker Compose + production Helm chart)**; no task timeouts, checkpoint/resume fits agent work; used by Cal.com, Midday, MagicSchool AI (https://trigger.dev/launchweek/2/trigger-v4-ga, https://trigger.dev/pricing, https://trigger.dev/docs/self-hosting/overview) [H pricing/GA, M customers].
- **Inngest 4.11.0** — event-driven durable steps, agent primitives; platform not self-hostable per third-party sources (unverified against current official docs) [M]; step-based pricing compounds on chatty LLM workflows.
- **Temporal** (TS SDK 1.19.0) — durability gold standard, highest conceptual load (deterministic workflows, worker versioning); reported $300M at $5B Feb 2026 [M — aggregator]. **Restate** promising, smaller ecosystem. **Cloudflare Workflows GA** Apr 2025 — only if all-in Workers [H].

**Recommendation:** **Trigger.dev v4** as the primary job/agent-execution substrate (TS-native durable execution, self-host exit hatch, direct comparables use it) — this doubles as the durable-execution layer §6 requires for AI workers. pg-boss for strictly-minimal short idempotent MVP jobs if you want zero extra vendors. Temporal only when workflows span days across services and a team can own it; BullMQ only for a raw-throughput hot path.

## 5. Auth for B2B SaaS

- **better-auth 1.6.x** (1.6.23, ~late Jun 2026; no 2.x — npm-verified) [H]. YC X25, **$5M seed Jun 2025** (Peak XV) (https://techcrunch.com/2025/06/25/this-self-taught-ethiopian-dev-built-an-authentication-tool-and-got-into-yc/) [H]. **Auth.js is now part of Better Auth** (Sept 26, 2025 — maintainer transition; Auth.js gets security-fixes-only, new projects pointed at better-auth: https://github.com/nextauthjs/next-auth/discussions/13252) [H]. Mature **organization plugin** (orgs, teams, invitations, custom + dynamic per-org roles — https://better-auth.com/docs/plugins/organization); **SSO plugin** covers OIDC/OAuth2/SAML 2.0 with recent replay-protection hardening (https://better-auth.com/docs/plugins/sso); **SCIM 2.0 server plugin now exists** (shipped after being a known 2025 gap — https://better-auth.com/docs/plugins/scim) [H exists, M maturity]. Paid "Better Auth Infrastructure" adds self-service SSO/SCIM onboarding UI, bot protection, log drains (pricing unpublished) [H exists, L pricing]. Recommended by Next.js/Nuxt/Astro docs; the 2026 default in new TS templates [M].
- **Auth.js/NextAuth:** maintenance mode; **v5 stays in beta indefinitely** [H]. Do not start new products on it. **The "WorkOS acquired Auth.js" claim is REFUTED** — the real event was the Better Auth transition [H].
- **Clerk:** $50M Series C Oct 2025 (Menlo + Anthropic's Anthology Fund — https://clerk.com/blog/series-c) [H]. ### RAG (2026 consensus)

Naive RAG is dead as a *default*, alive as a component. Pipeline: **hybrid retrieval (BM25 + dense, top 50–150) → cross-encoder rerank → top 5–8 to the model**; hybrid+rerank benchmarks ~+37% NDCG vs vector-only [M — second-hand benchmark]. **Anthropic contextual retrieval** (LLM-generated chunk context prepended before indexing): −49% retrieval failure with contextual BM25, **−67% with reranking**, ~$1.02/M doc tokens with prompt caching (https://www.anthropic.com/news/contextual-retrieval) [H, primary]. Chunking default: recursive 256–512 tokens, 10–20% overlap; late chunking as the embedding-side alternative [M-H]. Rerankers: **Cohere Rerank 4.0 shipped** (3.5 remains the battle-tested default — https://docs.cohere.com/docs/rerank) [H]; Voyage rerank-2.5 competitive [vendor-claimed, L-M]. Embeddings: `text-embedding-3-large` safe default; Voyage-3.5 quality leader for technical corpora; benchmark on your own data [M]. **Agentic retrieval:** retrieval becomes a tool the agent calls iteratively (query decomposition, search-during-think); keep one-shot RAG for latency-sensitive Q&A (https://www.llamaindex.ai/blog/rag-is-dead-long-live-agentic-retrieval) [H direction]. Vector store: per §4 — pgvector + RLS now, Turbopuffer namespace-per-tenant at ~10M+ vectors.

**Recommendations:** AI SDK v7 core; hand-rolled/`ToolLoopAgent` loops per Anthropic guidance; Claude Agent SDK evaluated for heavyweight "AI worker" tasks; Trigger.dev durability under every agent; MCP server as public surface; contextual-retrieval + hybrid + rerank RAG with retrieval exposed as an agent tool (hybrid + keyword + metadata-filter variants).

## 8. Realtime

- **DIY:** one Node WS service (`ws`, or Socket.IO — still alive and mainstream, ~5.3M wk downloads [L-M] — for free rooms/reconnect) + Redis pub/sub + sticky sessions past one instance. uWebSockets.js's 5–10x throughput (https://evilmartians.com/chronicles/choose-your-fighter-benchmarking-5-websocket-servers-for-nodejs) is overkill for a CRM-class app. At B2B SaaS scale (thousands of concurrent connections, not millions), this is ~a week of work and boring [M-H].
- **Managed:** **PartyKit was acquired by Cloudflare (Apr 2024)**, now deploys to your own CF account as Durable-Objects-with-a-nice-API — very cheap actor-per-room, ideal for presence (https://blog.cloudflare.com/cloudflare-acquires-partykit/) [H]. Liveblocks = product features (comments/presence/notifications) at MAU pricing that stings for daily-active B2B (Pro $25/mo+, Team $500/mo — https://liveblocks.io/pricing) [H]. Ably > Pusher above ~100k connections; both are transport, not data-plane [M — vendor comparison].
- **CDC as architecture:** Postgres logical replication → fan-out is now the dominant pattern (Electric shapes over HTTP; Supabase Realtime's WAL pipeline — https://supabase.com/docs/guides/realtime/architecture; Zero's replica). Virtue: the DB is the source of truth, invalidation can't be "forgotten." Costs: replication-slot ops (WAL retention, slot failover), and ephemeral presence doesn't belong in it [H synthesis].
- **Reference class:** Linear runs its own WS delta broadcast; Trigger.dev runs Electric CDC; Notion WS + local SQLite. **Nobody reference-class runs their core data plane on Pusher/Ably** — managed transports appear only for presence/cursors [H/M].

**Recommendation:** MVP: own thin WS service publishing coarse per-workspace invalidation events ("record X changed") consumed by TanStack Query/DB → refetch; Redis pub/sub for fan-out. Presence/cursors/typing on PartyKit/Durable Objects (off the DB path). Scale: replace hand-published invalidation with **CDC → ElectricSQL shapes**, which is exactly the §2 sync-engine upgrade path — the same bet twice, which is why the whole stack is coherent.

## Cross-cutting coherence notes

1. **Postgres is the center of gravity**: tenancy (RLS), vectors (pgvector), sync (CDC/Electric/Zero all require it), audit logs, and optionally jobs (pg-boss) all compound on one operational competence. Every recommended component keeps `tenant_id`-on-every-row discipline, which preserves all scale exits (Citus, PlanetScale, silo-per-enterprise-tenant).
2. **One durable-execution substrate** (Trigger.dev v4) serves both background jobs and AI-agent workers — per Anthropic guidance, agent side effects should be journaled durable steps anyway.
3. **The sync-engine and realtime bets are the same bet** (CDC via Electric + TanStack DB), adopted incrementally: Query + WS invalidation → DB collections → Electric shapes.
4. **Vendor-risk watchlist for ADRs:** better-auth (seed-stage), Drizzle 1.0 (RC), TanStack DB (beta), TanStack Start (RC), Zero (no public logos), Nile (preview), Neon (Databricks alignment), Turborepo 3.0 (telegraphed), Bun-as-runtime (avoid for now), Stainless (dead to new users post-Anthropic acquisition).
