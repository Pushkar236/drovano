# ADR-0005: API strategy — tRPC internal; REST/OpenAPI + SDKs + MCP public

- **Status:** Accepted
- **Date:** 2026-07-07
- **Deciders:** CTO
- **Tags:** backend, ai

## Problem

Drovano has two API audiences with opposite needs: the first-party SPA
(iteration speed, zero contract drift) and the public platform surface
(stability, discoverability, SDKs, agent-legibility — PRD §3.6 requires
REST + webhooks + MCP). Choose the protocol(s), schema source of truth,
and SDK pipeline. Forces: Zod is already the validation standard
(CODING_STANDARDS); the 2026 SDK-generation market just consolidated;
MCP became the integration surface.

## Alternatives considered

### Option A — tRPC v11 internal + REST/OpenAPI generated from Zod v4 (`@hono/zod-openapi`) public + Speakeasy SDKs + tenant-scoped OAuth MCP server

- tRPC v11: stable since Mar 2025, battle-tested (Cal.com), end-to-end
  types with zero codegen for the dashboard.
- Public REST generated from the **same Zod schemas** the services
  validate with — one source of truth, two surfaces.
- Speakeasy: the remaining major independent SDK generator (Stainless
  acquired by Anthropic May 2026, hosted generator winding down; Fern
  acquired by Postman); Dub's Node SDK is Speakeasy-generated.
- Weaknesses: two API styles to maintain; tRPC's own OpenAPI story is
  weak (why the public surface is REST-first, not tRPC-derived).
- Evidence: research §3 (versions verified 2026-07-06: `@trpc/server`
  11.18.0, Zod 4.4.3); Midday runs exactly this cohabitation.

### Option B — oRPC everywhere

- tRPC-class DX with first-class OpenAPI from the same procedures — one
  system for both surfaces; real traction (615k weekly downloads ~15
  months post-1.0).
- Weaknesses: an order of magnitude less battle-testing than tRPC;
  betting both surfaces on one young project.

### Option C — REST/OpenAPI only (no RPC layer)

- One style, maximal simplicity. But the dashboard pays contract-drift
  and boilerplate tax on every feature; type-safety via codegen loops
  instead of inference.

### Option D — GraphQL

- Flexible queries for a graph-shaped product (tempting on paper).
- Weaknesses: declining as a default for single-first-party-client
  products; 0/5 reference repos use it for core APIs; resolver-layer
  N+1/authz complexity directly conflicts with the RLS/permission-service
  design; federation solves problems we don't have.

## Research

`docs/research/technology-stack-2026.md` §3 and §6 (MCP). SDK-market
events: Stainless→Anthropic (2026-05-18), Fern→Postman (2026-01-08),
sources in research. MCP: donated to Linux Foundation Dec 2025; target
the stateless 2026-07-28 spec revision; precedent for OAuth MCP servers:
Stripe, Linear, Cloudflare.

## Decision

tRPC v11 for the dashboard↔backend contract; public surface is REST with
OpenAPI generated from the shared Zod v4 schemas via `@hono/zod-openapi`;
customer SDKs (TypeScript first) generated by Speakeasy; a tenant-scoped
OAuth MCP server wraps the public operations, targeting the stateless
2026-07-28 MCP revision.

## Why this option

1. Each audience gets the right tool without inventing anything: zero
   drift internally, stable contract publicly.
2. Zod schemas as the single source of truth means the public API is a
   projection of the same validated types the product runs on — API docs
   can't lie.
3. MCP server = agent-legible in both directions (PROJECT.md law 2 and
   the researched "AI-native pattern 7") and free distribution inside
   Claude/ChatGPT/Copilot.
4. Speakeasy is the only SDK pick that survived 2026's consolidation.

## Trade-offs accepted

- Two styles to document and test (mitigated: same services, same
  schemas underneath; the public surface is deliberately smaller).
- Public API versioning discipline required from its first release
  (v1 path prefix; additive-only within a version).
- Speakeasy is a paid dependency (acceptable; SDK quality is
  platform-credibility).

## Future impact

- Easier: marketplace/embedded apps later (public surface already
  first-class); agent integrations; API-first enterprise deals.
- Harder: exposing internal-only tRPC procedures publicly requires a
  deliberate promotion step (feature, not bug).
- Revisit: oRPC after another year of production maturity (could unify
  both surfaces); MCP spec revisions (track the Agentic AI Foundation
  release train); if the public API needs GraphQL-shaped querying,
  consider a read-only query endpoint rather than a GraphQL layer.
